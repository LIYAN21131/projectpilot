# ProjectPilot 简历表达质量评分系统设计

日期：2026-06-24  
状态：已确认，待实施计划

## 1. 目标与边界

在现有流程中增加简历表达质量评分，让用户对比 AI 识别并确认后的原始内容与 AI 优化后的简历内容，直观看到表达变化。

现有核心流程保持不变：

> 用户输入项目资料 → AI 识别结构化内容 → 用户确认保存 → 简历优化 → 面试题生成与回答 → 复制/保存结果

评分系统只评估表达质量，不判断经历真实性。前端不展示真实性风险等级或负面警告，但所有评分和优化 Prompt 必须禁止虚构事实、经历和数据。

评分属于辅助能力。评分失败不得阻断项目保存、简历优化、复制、面试准备或其他已有流程。

## 2. 总体方案

采用分阶段混合方案：

1. 用户确认 AI 识别结果后，项目照常保存，并异步生成原始评分。
2. 简历优化继续使用现有接口和业务逻辑。
3. 优化完成后，单独评估优化文本。
4. 优化后评估服务同时接收原始内容、原始评分和优化内容，返回优化后维度评分及前后对比总结。
5. 总分、维度变化和总提升值由程序计算，不接受 AI 提供的算术结果。
6. 评分结果保存在对应 `Project` 对象中。

主展示入口位于项目编辑页的“简历优化”步骤。MVP 不在独立 `/resume` 页面维护另一套完整生成和保存流程。

## 3. 流程节点

### 3.1 原始评分

触发时机：用户点击“确认保存识别结果”，项目确认数据成功保存之后。

评分输入：

- `confirmedResumeFields`
- 当前目标岗位 `targetRole`
- 固定评分标准版本

处理规则：

- 评分请求异步执行，不阻断保存成功反馈。
- 成功后将 `before` 评分写入项目。
- 失败时保留项目数据，并在评分区提供重试。

### 3.2 优化后评分

触发时机：现有简历优化成功并得到非空优化文本之后。

评分输入：

- 已确认的原始简历字段
- 原始评分
- 优化后的简历 bullet
- 当前目标岗位
- 与原始评分相同的评分标准版本

处理规则：

- 现有优化文本立即正常展示。
- 评分请求失败不影响复制和保存。
- 成功后生成 `after` 评分与 `comparison` 总结。
- 优化后评分先保存在页面状态；用户点击现有“保存到项目”操作时，与优化文本一起写入项目。
- 优化后分数允许提升、持平或下降。

### 3.3 内容变化

如果已确认项目内容、优化文本、目标岗位或评分标准版本发生变化：

- 保留旧评分。
- 将评分状态标记为 `stale`。
- 展示“项目内容或目标岗位已更新，此评分对应上一版本”。
- 不自动消耗 AI 请求，由用户主动重新评分。

## 4. 评分标准

总分 100 分，五个维度等权，每项 20 分。

| Key | 展示名称 | 评估范围 |
|---|---|---|
| `completeness` | 信息完整度 | 是否交代项目背景、目标或任务、关键行动和结果 |
| `impact` | 成果表达力 | 是否清楚说明项目产出或价值；无数据时评估定性结果是否清楚 |
| `logic` | 逻辑清晰度 | 信息顺序、因果关系及 STAR 或“背景—行动—结果”结构是否明确 |
| `roleFit` | 岗位匹配度 | 是否体现目标岗位能力；产品经理重点关注需求分析、用户研究、方案设计、数据意识、协作推动、落地执行和复盘 |
| `professionalism` | 表达专业度 | 是否简洁、具体、专业，避免口语化、流水账、重复和空泛表达 |

### 4.1 分数区间

- `0–7`：信息严重不足或难以理解。
- `8–12`：基本可读，但存在明显缺项或表达松散。
- `13–16`：内容较完整，达到简历可用水平。
- `17–18`：结构清楚，岗位表达较强。
- `19–20`：仅用于证据充分、逻辑严密且表达成熟的内容。

### 4.2 评分约束

- 每项必须是 `0–20` 的整数。
- 优化前后必须使用同一套 rubric。
- 不把量化数据作为高分的强制条件。
- 缺少事实时可因信息不完整扣分，但不得建议或生成虚假事实。
- 评分必须有区分度，不得普遍给高分。
- 优化后的变化必须能从前后文本中找到依据。

## 5. AI 输出合同

AI 返回单项维度分数、理由和总结。程序负责计算总分与差值。

### 5.1 单份评分

```json
{
  "dimensions": [
    {
      "key": "completeness",
      "score": 12,
      "reason": "已说明项目背景和职责，但行动过程与结果信息较少。"
    },
    {
      "key": "impact",
      "score": 10,
      "reason": "表达了项目产出，但尚未清楚说明产生的具体价值。"
    },
    {
      "key": "logic",
      "score": 13,
      "reason": "主要信息可读，但背景、行动和结果之间的衔接仍较弱。"
    },
    {
      "key": "roleFit",
      "score": 14,
      "reason": "体现了部分产品工作，但关键决策和推动过程不够突出。"
    },
    {
      "key": "professionalism",
      "score": 13,
      "reason": "语言基本专业，但仍有任务罗列和重复表达。"
    }
  ],
  "summary": "内容基本完整，但行动与成果之间的联系还不够清楚。"
}
```

### 5.2 优化后评估与对比总结

```json
{
  "dimensions": [
    {
      "key": "completeness",
      "score": 18,
      "reason": "优化后形成了背景、行动和结果的完整链路。"
    }
  ],
  "summary": "项目逻辑与岗位相关能力表达更加集中。",
  "comparison": {
    "dimensionReasons": [
      {
        "key": "completeness",
        "reason": "补清了行动过程和结果之间的关系。"
      }
    ],
    "highlights": [
      "项目目标更加明确",
      "行动描述从任务罗列调整为具体推进过程",
      "成果与产品经理岗位能力的关联更清楚"
    ],
    "summary": "优化后表达更完整、清晰，更适合产品经理简历场景。"
  }
}
```

### 5.3 服务端规范化

服务端必须：

- 校验五个固定 key 均存在且不重复。
- 将分数规范为 `0–20` 整数。
- 拒绝缺失维度或结构非法的结果。
- 将理由控制为非空字符串。
- 将 highlights 限制为最多三条。
- 根据维度分数计算总分。
- 根据前后分数计算每个维度变化和总提升值。
- 第一次响应结构无效时允许一次受控重试。

## 6. AI 限制 Prompt

```text
你是中文产品岗位简历表达质量评估器。你的任务是评估文本的表达质量，
而不是判断经历真实性，也不是替用户补写不存在的事实。

必须遵守以下规则：

1. 不得虚构用户未提供的数据、指标、人数、金额、比例、转化率或业务结果。
2. 不得编造不存在的公司、项目、岗位、工具、职责、行动或经历。
3. 可以调整措辞、结构和信息顺序，但不能改变原始事实含义。
4. 如果内容缺少量化数据，只能评价现有定性成果是否表达清楚；
   不得因为没有数字而强制判定为低质量，也不得凭空补充数字。
5. 评分只评估简历表达质量，不评估经历是否真实，不输出真实性判断。
6. 评分应结合目标岗位要求。产品经理岗位重点关注需求分析、用户研究、
   产品设计、数据意识、协作推动、落地执行和复盘优化。
7. 必须使用给定的五个评分维度和评分区间，每项返回 0 到 20 的整数。
8. 分数必须有区分度。普通可用内容通常为 12 到 16 分，
   19 到 20 分只用于信息充分、逻辑严密且表达成熟的内容。
9. 优化后评分允许持平或下降，不得为了展示优化效果强行提高分数。
10. 每项评分理由必须能从输入文本中找到依据。
11. 对比总结只能描述优化前后真实存在的文本变化。
12. 如果优化文本引入原文没有的新事实，不得将该内容视为加分依据。
13. 输出必须是合法 JSON，不得输出 Markdown、代码围栏或解释性文字。
14. 不得输出风险等级、真实性风险提示或可能引发用户焦虑的措辞。
15. reason、summary 和 highlight 应简洁、专业、中性，不使用营销语言。
```

具体请求必须同时提供固定 JSON Schema、目标岗位、评分 rubric 和待评估文本。

## 7. 数据模型与持久化

不新增 localStorage key，扩展现有 `Project`：

```ts
type ResumeQualityDimensionKey =
  | "completeness"
  | "impact"
  | "logic"
  | "roleFit"
  | "professionalism";

type ResumeQualityDimensionScore = {
  key: ResumeQualityDimensionKey;
  name: string;
  score: number;
  reason: string;
};

type ResumeQualityScore = {
  total: number;
  dimensions: ResumeQualityDimensionScore[];
  summary: string;
};

type ResumeQualityDimensionChange = {
  key: ResumeQualityDimensionKey;
  before: number;
  after: number;
  change: number;
  reason: string;
};

type ResumeQualityComparison = {
  improvement: number;
  dimensionChanges: ResumeQualityDimensionChange[];
  highlights: string[];
  summary: string;
};

type ResumeQualityAssessment = {
  version: 1;
  rubricVersion: 1;
  targetRole: string;
  before?: ResumeQualityScore;
  after?: ResumeQualityScore;
  comparison?: ResumeQualityComparison;
  status: "current" | "stale";
  sourceFingerprint: string;
  createdAt: string;
  updatedAt: string;
};
```

`Project` 新增：

```ts
resumeQualityAssessment?: ResumeQualityAssessment;
```

`sourceFingerprint` 由以下内容的稳定序列化结果生成：

- `confirmedResumeFields`
- `optimizedResumeBullets`
- `targetRole`
- `rubricVersion`

MVP 可使用浏览器原生稳定哈希或确定性字符串摘要，不引入额外依赖。

原始评分阶段的 fingerprint 只包含 `confirmedResumeFields`、`targetRole` 和
`rubricVersion`。完整对比阶段再加入 `optimizedResumeBullets`。组件应按当前阶段
计算期望 fingerprint；不匹配时将已有结果标记为 stale。

## 8. 前端展示

评分模块位于项目编辑页“步骤 3：生成简历表述”中，建议放在优化文本之后、保存操作附近。

### 8.2 展示规则

- 顶部展示优化前、优化后和变化三个数字。
- 正向变化显示 `+N`，持平显示 `0`，下降显示 `-N`。
- 下降使用中性文案，不使用警告图标或风险措辞。
- 五个维度使用简单双进度条或对比列表，不使用雷达图。
- 维度理由默认收起，可按需展开。
- 主要提升最多三条。
- 原始评分生成完成但尚未优化时，只显示优化前分数与“完成简历优化后查看对比”。
- 加载状态使用局部骨架或轻量 loading，不遮挡已有内容。
- 错误状态显示“评分暂未生成”和重试按钮。
- stale 状态保留旧结果，并提示评分对应上一版本。

## 9. 模块边界

为避免继续扩大现有 `ProjectEditor.tsx`，实现时应采用聚焦拆分，不进行无关重构：

- `types/resume-quality.ts`：评分类型与固定维度 key。
- `lib/resume-quality/normalize.ts`：AI 响应校验、分数规范化、总分和变化计算。
- `lib/resume-quality/fingerprint.ts`：稳定内容指纹。
- 评分 API route：处理 DeepSeek 请求、Prompt、Schema 校验和受控重试。
- `lib/ai/client.ts`：新增原始评分和优化后评分客户端函数。
- `components/resume/ResumeQualityComparison.tsx`：纯展示组件。
- `ProjectEditor.tsx`：仅负责触发、状态协调和持久化。

评分展示组件只依赖规范化后的 `ResumeQualityAssessment`，不直接调用 AI 或 localStorage。

## 10. 异常处理

| 情况 | 行为 |
|---|---|
| 原始评分失败 | 项目照常保存；显示轻量错误状态和重试 |
| 优化后评分失败 | 优化文本正常展示、复制和保存；对比区单独重试 |
| AI 返回非法 JSON | 不保存半成品；执行一次受控重试 |
| 分数越界 | 服务端规范为 `0–20` 整数 |
| 维度缺失或重复 | 响应无效；重试后仍失败则返回错误 |
| 前后 rubric 版本不同 | 不计算对比，要求重新生成原始评分 |
| 优化文本为空 | 不发起优化后评分 |
| 内容或岗位变化 | 保留旧评分并标记 stale |
| 页面刷新 | 从项目对象恢复评分和状态 |
| DeepSeek key 缺失或服务不可用 | 评分失败但不影响核心流程；MVP 不伪造本地评分结果 |

最后一条与现有简历优化 fallback 不同：评分系统若没有可靠 AI 结果，应显示未生成，不应使用规则拼出看似精确的分数。

尚未点击“保存到项目”的优化文本及优化后评分属于页面临时状态，刷新后可丢失；
已保存的原始评分和完整对比结果必须能够恢复。

## 11. 测试策略

### 11.1 纯逻辑测试

- 五个维度总分计算正确。
- 正数、零和负数变化计算正确。
- 分数越界和小数规范化正确。
- 缺失、重复和未知维度被拒绝。
- highlights 被限制为三条。
- 相同输入生成相同 fingerprint。
- 项目内容、岗位、优化文本或 rubric 变化会改变 fingerprint。

### 11.2 API 测试

- Prompt 包含事实限制规则。
- 合法响应被规范化。
- 非法 JSON 触发一次重试。
- 重试失败返回明确错误。
- 缺少输入返回 400。
- 服务不可用不写入虚假评分。

### 11.3 流程测试

- 确认识别结果后项目先保存，再异步评分。
- 评分失败不阻断简历优化。
- 优化成功后文本先展示，评分独立加载。
- 评分失败不影响复制和保存。
- 保存后刷新页面可恢复评分；未保存的优化后评分不承诺恢复。
- 编辑项目或切换岗位后旧评分变为 stale。
- 持平和下降结果能够正常展示。

## 12. MVP 范围

### 包含

- 固定五维等权评分。
- 确认识别后自动生成原始评分。
- 优化完成后生成优化评分和对比总结。
- 项目编辑页展示总分、维度对比和三条主要提升。
- 评分保存到现有项目对象。
- loading、失败、重试和 stale 状态。
- 服务端 JSON 校验与事实限制 Prompt。
- 程序计算总分与差值。
- 优化后允许持平或下降。

### 不包含

- 雷达图、趋势图和历史版本曲线。
- 用户自定义权重。
- 多岗位同时评分。
- 评分排行榜、等级徽章或庆祝动画。
- 对外分享评分报告。
- 独立 `/resume` 页完整接入。
- 自动循环优化直到分数提升。
- 真实性风险 UI。

## 13. 后续迭代

- 保存评分历史并比较不同简历版本。
- 切换目标岗位后重新评分。
- 点击维度定位对应文本。
- 根据低分维度执行定向重写。
- 支持采纳或忽略单条优化建议。
- 监控评分稳定性并支持 rubric 版本迁移。
- 在独立简历页读取项目评分。
- 增加运营、数据分析、设计和研发岗位 rubric。

## 14. 验收标准

- 用户确认识别内容后，无需额外操作即可得到原始评分；失败时不影响项目保存。
- 用户完成简历优化后可看到优化前、优化后、总变化、五维变化和最多三条提升点。
- 所有分数均来自固定五维 rubric，且由程序计算总分和变化。
- 优化后分数可提升、持平或下降。
- 评分结果可随项目保存并在刷新后恢复。
- 内容或岗位变化后旧评分保留但明确标记 stale。
- AI 不得虚构数据、事实、经历、岗位、工具、职责或结果。
- 页面不展示真实性风险等级或引发焦虑的提示。
- 评分系统失败不影响识别、保存、优化、复制、面试生成与回答。
